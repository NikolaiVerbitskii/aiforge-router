Cursor discovers hybrid-mcp through .cursor/mcp.json. The agent (Claude) sees the registered tools and their descriptions, then autonomously decides when to call them — no explicit prompt injection needed if descriptions are clear.
.cursor/mcp.json
{ "mcpServers": { "hybrid-mcp": { "command": "node", "args": ["./hybrid-mcp/dist/index.js"], "env": { "OLLAMA_HOST": "http://localhost:11434" } } } }
.cursor/rules (optional — guides when to use hybrid-mcp)
# cursor rules When performing code analysis, refactoring, test generation, or explanation tasks — prefer using the `hybrid-mcp` run_task tool first. It processes locally (private, fast, no cloud cost). Only handle the result directly with your own reasoning when: - The local result has low confidence (< 0.7) - The task requires understanding the full project architecture - The user has explicitly asked for your direct analysis
Trigger: when does Cursor call hybrid-mcp?
Cursor agent (Claude) reads the tool description and decides. Two scenarios: 1. AUTONOMOUS — Claude reads the tool description: "Processes coding tasks locally on private hardware. Use for: refactor, explain, test, debug, analyze. Returns structured result. Local = private + fast." → Claude chooses it for appropriate tasks automatically. 2. RULE-GUIDED — .cursor/rules adds explicit preference: "prefer hybrid-mcp for code analysis tasks" → Cursor follows the rule before falling back to own reasoning.
This is exactly what Cursor's Claude "sees" — the tool name and description determine when Claude decides to use it. Get the description right and Claude routes correctly without any extra prompting.
